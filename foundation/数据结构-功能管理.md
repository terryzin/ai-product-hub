# 数据结构 · 功能管理

> **目的：** 从现有《SimOne功能版本管理》Excel 中**抽象出它到底管了什么、需要哪些数据结构来描述**，作为新平台的数据模型设计输入。方法是先做全列取值剖析（枚举/填充率/基数），再还原每列的真实语义，最后归并成实体模型。
>
> **一句话结论：** 这张 484 行的表，表面是"一张功能清单"，实际上**挤压了 5 个不同实体**进一个二维网格：功能条目、变体适用性、发布版本、项目定制、晋升状态语义。新平台要做的第一件事就是**把它们拆成正交的结构**。

---

## 1. 它到底管了什么（剖析结论）

| 源工作表 | 行数 | 实际承载的实体 |
|---|---|---|
| 功能管理基础表 | 484 | 功能条目 + 变体适用性 + 项目定制（三者被压在一张表里） |
| 版本管理 | 26 | 发布版本（独立实体） |
| 填写说明 | 5 | 单元格取值的**语义图例**（晋升生命周期） |

剖析得到的关键事实：

- **层级是四级 + 一个正交标签：** `总分类`(14) → `功能模块`(35) → `一级功能`(125) → `二级功能`(226，含 N/A)，另有 `业务支撑`(40) 作为跨层的业务能力标签。
- **6 个变体列不是布尔，而是四态多态值**（见 §3）——这是全表信息密度最高、也最被低估的地方。
- **`项目&版本号` 列把"项目定制"塞进了一个自由文本格**：长城(130)/几何(25)/江淮(14)/潍柴/劢擎/奔驰/地平线/同济… 约 19 个客户，且有脏数据（`江淮3.5` vs `江淮-协作版3.5` vs `数智交院-单机版3.43.4`）。
- **`填写说明` 揭示了 是/否/空白 的真实语义**：不只是"适不适用"，还编码了**项目定制的晋升状态**。

## 2. 实体模型（抽象后的目标结构）

把压在一张表里的东西拆开，得到 5 个实体：

```
┌─────────────────────────────────────────────────────────────┐
│  Feature 功能条目（核心）                                       │
│  id · 层级(area/module/l1/l2) · 描述 · 能力标签 · 文档 · 备注    │
│  · 客户展示 · 审计(编辑人/时间)                                  │
└───────┬─────────────────────────────┬───────────────────────┘
        │ 1 : N                        │ 1 : N
        ▼                              ▼
┌───────────────────────┐   ┌──────────────────────────────┐
│ VariantApplicability   │   │ ProjectCustomization 项目定制   │
│ 变体适用性（每功能×每单元）│   │ feature × 客户/项目 × 版别版本   │
│ 四态：见 §3            │   │ + 晋升状态(§4)                 │
└───────────┬───────────┘   └──────────────┬───────────────┘
            │ 引用                           │ 引用
            ▼                                ▼
┌───────────────────────┐   ┌──────────────────────────────┐
│ VariantCell 变体单元    │   │ Project 项目/客户线             │
│ = 商业版 × 形态（受控词表）│   │ 长城/几何/江淮…（应建为实体）     │
└───────────────────────┘   └──────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ Release 发布版本（独立）：版别 × OS × 版本号 × 状态 × 制品        │
└─────────────────────────────────────────────────────────────┘
```

## 3. 核心发现：变体适用性是"四态多态值"，不是布尔

6 个变体列（`单机版`/`协作版_客户端`/`协作版_云端`/`教学-单机版`/`科研-单机版`/`教学-协作版`）的取值剖析显示，一个单元格可能是四种语义之一：

| 形态 | 真实取值（来自剖析） | 语义 | 目标数据结构 |
|---|---|---|---|
| **适用性 presence** | `是` / `否` | 该变体单元支持/不支持 | `presence`（default-on，仅标例外） |
| **参数 parameter** | `限50个` / `限4个` / `限23个` / `限500个` / `无限制` | 同一功能、不同单元的**数量上限差异** | `parameter.by_cell: {teach_standalone: 50, …}` |
| **行为差异 override** | `不限制登录，仅限制创建任务` | 同一功能、某单元的**散文级行为 delta** | `override[].delta` |
| **继承 inherit** | `与协作版一致` | 不复制，指向另一单元 | `override[].inherit_from` |

> 📌 **这正是本仓库 `presence` / `parameter` / `override(delta/inherit_from)` 模型的现实数据源。** Excel 已经在用这套语义，只是把四种东西混在一个文本格里、靠人读。新平台要做的是**把这四态显式化、结构化、可机检**。

### 字段级 schema：Feature

| 字段 | 源列 | 类型 | 枚举/约束 | 对应仓库概念 |
|---|---|---|---|---|
| id | 自动编号 | 稳定 ID | 唯一，永不复用 | `id: F-xxxx` |
| area 区域 | 总分类 | enum(受控) | 用例/测试/主车/资源/后台/可视化/顶部/算法/DIL/合成数据/架构/API… | `tags.area`（taxonomy.yaml） |
| module 模块 | 功能模块 | enum(受控) | 35 项 | `tags.module` |
| title_l1 | 一级功能 | string | 必填 | `title`（level=feature） |
| title_l2 | 二级功能 | string\|N/A | N/A=只有一级 | 子项（level=subitem） |
| capability 业务能力 | 业务支撑 | enum(多值,受控) | 40 项 | `tags.capabilities[]` |
| contract 描述 | 功能描述 | text | 一句话契约 | `interface.contract` / 正文 |
| docs 相关文档 | 相关文档 | link[] | 富文本/外链 | 正文引用 |
| note 备注 | 备注 | text | 常含 override delta 原话 | `override[].delta` 素材 |
| customer_facing | 客户展示 | bool | 是/否 | 渲染过滤标志 |
| impl_version 实现版本 | 实现版本 | semver | 3.4.0 居多 | `revision.version` |
| update_version 更新版本 | 更新版本 | semver | | 后续 revision |
| applicability | I~N 六列 | VariantApplicability[] | 见上表四态 | `presence`+`override`+`parameter` |
| project_custom | 项目&版本号 | ProjectCustomization[] | 见 §4 | `scope`+`product-lines` |
| audit | 最后编辑人/创建时间/最后编辑时间 | meta | | Git 留痕 |

## 4. 晋升状态语义（来自《填写说明》，极重要）

`填写说明` 把单元格取值定义成了**项目定制 → 标准版的晋升生命周期**——这远超"适用性"，是 `scope` 的现实依据：

| 取值 | 含义（原文） | 映射到 scope/生命周期 |
|---|---|---|
| `否` | 标准版本不支持 | 不适用（presence.exclude） |
| `是` | 标准版本已支持 | 核心（已进 baseline） |
| `是`（注） | 标准版还不支持，但项目定制已实现，**可对外宣传** | 可晋升 · 已实现待回收 |
| `是`（注） | 项目定制迭代优化，**与标准版有差异** | 可晋升 · 有 override delta |
| `空白` | 项目定制实现，**短期不进标准版、不对外宣传** | 客户独占（禁止进 baseline） |

> 这张图例说明：**"是/否/空白" 三值其实承载了 scope（核心/可晋升/客户独占）+ 宣传可见性 + 迁移待办**。新平台必须把这条隐含的生命周期显式建模，否则"哪些该合回基线"永远靠人读备注猜（呼应痛点 R2/R4）。

### 字段级 schema：ProjectCustomization

| 字段 | 源 | 类型 | 说明 |
|---|---|---|---|
| feature_ref | 行 | FK→Feature | 属于哪个功能 |
| project 客户/项目 | 项目&版本号（拆分） | FK→Project | 长城/几何/江淮…需建 Project 实体 |
| edition_version | 项目&版本号（拆分） | string | 如 `协作版3.6` |
| promotion_status | 单元格取值+图例 | enum | 核心/可晋升-已实现/可晋升-有差异/客户独占 |
| public_visible | 图例 | bool | 是否可对外宣传 |

## 5. 发布版本（Release，独立实体）

`版本管理` 表是与功能正交的**发布台账**，26 行：

| 字段 | 源列 | 类型 | 枚举 |
|---|---|---|---|
| edition 版别 | 版本 | enum | 企业版-单机 / 企业版-协作 / 企业版-多车 / 教育版-教学 / 教育版-科研 |
| version 版本号 | 版本号 | string | 3.4.0~3.8.2，含 Worldeditor 等特例 |
| os | OS | enum | Windows / Linux |
| release_status 发布状态 | 发布状态 | enum | 已发布 / 发布中 / 未开始 |
| released_at | 发布时间 | date | |
| artifacts 制品 | NAS / 百度网盘 | link[] | 产物存放位置 |
| audit | 创建人/编辑人/时间 | meta | |

> Release 的 `edition` 与功能表的 6 个变体列、资产表的版别列**是同一维度的三种不同切法**——见 §6 的关键问题。

## 6. 关键发现：变体维度有"三套不一致的词表"

同一个"商业版 × 形态"维度，在三处用了**不同的取值集**：

| 上下文 | 用的变体取值 |
|---|---|
| 功能基础表（6 列） | 单机版 / 协作版_客户端 / 协作版_云端 / 教学-单机版 / 科研-单机版 / 教学-协作版 |
| 版本管理（版别） | 企业版-单机 / 企业版-协作 / 企业版-多车 / 教育版-教学 / 教育版-科研 |
| 资产表（版别列） | 公有云 / 社区版 / 协作版 / 企业版-单机 / 企业版-云端 / 教育版（含教学版）/ 企业版⭐科研版 / 教学版 |

**这三套对不上**（如功能表有"协作版云端/客户端"细分，版本管理只有"协作"，资产表又冒出"公有云/社区版"）。这正是痛点 R1/R2 的微观证据，也是新平台必须先立**统一受控词表**的硬理由：

> **设计要求 D1：** 维度 = `商业版 × 形态 × OS`，出货组合（variant_cell）必须来自单一 `dimensions.yaml`；上述三套切法须收敛映射到同一套 variant_cells（本仓库已有雏形）。

## 7. 抽象出的目标数据结构（汇总）

```yaml
Feature:                       # 功能条目（核心）
  id: F-0001                   # 稳定唯一
  area: enum                   # 总分类
  module: enum                 # 功能模块
  title_l1: string
  title_l2: string|null        # N/A → null
  capabilities: [enum]         # 业务支撑（多值）
  contract: text               # 功能描述
  docs: [link]
  customer_facing: bool
  revision: {impl_version, update_version}
  applicability:               # ← I~N 六列，四态
    presence: {only|exclude: [variant_cell]}
    parameter: {name, by_cell: {variant_cell: value}}   # 限N个
    override: [{cell, delta} | {cell, inherit_from}]     # 散文/继承
  audit: {editor, created_at, updated_at}

ProjectCustomization:          # 项目定制（一功能可多条）
  feature_ref: F-0001
  project_ref: 长城
  edition_version: 协作版3.6
  promotion_status: enum        # 核心|可晋升-已实现|可晋升-有差异|客户独占
  public_visible: bool

Project:                       # 客户/项目线（需新建为实体）
  id: 长城
  ...→ 对应 product-lines/*.yaml profile

Release:                       # 发布版本（独立台账）
  edition: enum
  version: semver
  os: enum
  status: enum
  released_at: date
  artifacts: [link]

# 受控词表（引擎）
Dimension:                     # 统一变体维度
  commercial: [公有云,社区版,企业版,教育版(教学/科研)]
  form_factor: [单机,协作(客户端/云端),云端,多车]
  os: [Windows, Linux]
  variant_cells: [...]          # 合法出货组合（单一来源）
```

## 8. 与本仓库现有模型的对应 & 差距

| 抽象实体 | 仓库现有 | 差距 |
|---|---|---|
| Feature 层级/标签 | features/*.md 头 + taxonomy.yaml | taxonomy 需补全 35 模块/40 能力 |
| 变体四态适用性 | presence/parameter/override | **模型已就位**，需把 484 行数据导入 |
| ProjectCustomization/晋升 | scope + product-lines profile | 晋升状态枚举需固化进 enums.yaml |
| Project 实体 | product-lines/长城.yaml 等 | 需为 19 个客户各建 profile |
| Release 发布台账 | （空白） | 新平台需新增 Release 实体 |
| 统一变体词表 | dimensions.yaml | 需收敛三套切法（D1） |
