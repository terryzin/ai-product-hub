# 用户画像 · 深度分析

> **方法论：** 不写"产品经理/研发/测试"这种职位标签式画像。真正要抽取的是——**谁因为什么深层动机而用它、他半夜睡不着在焦虑什么、成功对他意味着什么**。并且抓住这个产品最反常识的一点：**AI agent 是一等公民用户，不是被使用的工具。**

---

## 0. 最关键的洞察：用户是"人 + AI"双层结构

口述全程反复出现一个范式判断——"产品工作过去面向人，现在要面向 AI native"、"未来文档可能不写给人看，写给机器就行"、"让 agent 源源不断帮我处理工作"。

这意味着本产品的用户画像**不能只画人**。它有两层用户，且系统必须**同时**服务好两层：

```
       ┌─────────────────────────────────────────┐
       │            人类用户（决策与协作）           │
       │  P1 产品掌舵者(核心) · P2 需求供给者         │
       │  P3 下游执行者 · P4 项目交付者              │
       └───────────────┬─────────────────────────┘
                       │ 双向读写、协作
       ┌───────────────┴─────────────────────────┐
       │       AI Agent 用户（一等公民）            │
       │  A1 上游写入 agent · A2 下游消费 agent      │
       │  A3 维护治理 agent                         │
       └─────────────────────────────────────────┘
```

**设计含义：** 任何一个能力，都要同时回答"人怎么用"和"agent 怎么用同一份数据"。这是贯穿全部画像的第一原则。

## 人类用户

### P1 · 产品掌舵者（核心用户，口述者本人）

> 一句话：**对一个多业务域、多形态、标准+项目交织的复杂产品矩阵负全局责任的人**，需要在 AI 时代用极少的人力把整盘棋管清楚、并交给 agent 提速。

| 维度 | 内容 |
|---|---|
| **是谁** | 产品负责人/产品总监角色。同时对 SimOne、Dataverse 等多产品、多业务域负责。 |
| **核心职责** | 需求收集 → 需求池 → planning → 开发分配 → 发布 → 反馈，全流程对 feature 负责。 |
| **要做的事(JTBD)** | "当需求从四面八方涌来时，我要把它们汇成可信的池子、抽象成能落地的功能点、判断进基线还是进项目、并随时说清每个 feature 现在到底什么状态。" |
| **深层动机** | 不是"想要个工具"，而是**想夺回对复杂度的掌控权**——从"被矩阵和熵增推着走"变成"能一眼看全局、能放心授权"。 |
| **半夜的焦虑** | ①"我其实说不清现在到底是什么状态。" ②"哪些该合回基线、哪些不该，我讲不清楚。" ③"别人用 AI 把链路自动化了，我还在手工维护，会被甩开。" |
| **成功的样子** | 任意时刻能问出/被问到"X 现在什么状态、在哪些线、谁依赖它"并立刻得到可信答案；大量重复工作交给 agent；自己只做高价值决策。 |
| **失败/被惹恼** | 又一个"建了表但没人维护、最后变垃圾场"的工具；又一次"改了这没改那"导致交付翻车。 |

**P1 是核心。其余画像都围绕"让 P1 掌舵得动"而存在。**

### P2 · 需求供给者（上游，人）

> 售前、销售、产品支持、客户、老板……需求的来源极度分散。

| 维度 | 内容 |
|---|---|
| **是谁** | 一切会提出需求的人：售前/销售/PM 支持/客户/行业前瞻/老板。 |
| **要做的事(JTBD)** | "我在某个聊天/会议里冒出一个需求或反馈，我希望它**零摩擦**地进入池子、且不丢细节——我不想为了'录进系统'而专门去学一个工具、填一堆字段。" |
| **深层动机** | 让"我说过的话"被听见、被沉淀，而不是聊完即蒸发。 |
| **关键约束** | **他们不会主动维护。** 任何指望他们自律填表的设计都会失败（见痛点 R3）。所以系统对 P2 的正确假设是：**采集必须自动化、走他们已在用的 channel（企微/飞书机器人）**，而不是把他们请进一个新工具。 |
| **成功的样子** | 在原有的沟通工具里随口一说，需求就带着上下文进了池子。 |

### P3 · 下游执行者（人）

> 研发、测试——消费产品定义的人。

| 维度 | 内容 |
|---|---|
| **是谁** | 拿着 feature 定义去实现、去测试的人。 |
| **要做的事(JTBD)** | "我要拿到**当前、准确、无歧义**的 feature 定义和验收标准去干活，而不是面对一份不知道过期没有的文档。" |
| **深层动机** | 不想因为"文档和真相对不上"而返工、背锅。 |
| **痛点关联** | 直接受 R4（文档漂移）伤害——拿到漂移的定义，实现就错。 |
| **成功的样子** | 文档可信、可追溯到血缘；能区分"这是基线行为还是本项目 override"。 |

### P4 · 项目交付者（人）

> 在客户项目分支上做定制、并要交付手册的人。

| 维度 | 内容 |
|---|---|
| **是谁** | 项目经理/交付工程师，工作在某个客户的项目分支上。 |
| **要做的事(JTBD)** | "我基于基线做客户定制，要说清哪些是纯定制、哪些是通用未实现、哪些定制以后可能转标准；交付时还要给出**和本项目实际功能一致**的用户手册。" |
| **深层动机** | 既要满足这个客户，又不想给基线和未来埋债。 |
| **半夜的焦虑** | "客户在他的分支上越走越远、不回升，我的手册和文档永远追不上。" |
| **痛点关联** | R2（基线↔项目分叉）+ R4（手册漂移）的双重受害者。 |
| **成功的样子** | 定制以"profile + override"表达而非 fork；手册跟随 feature 自动同步/派生。 |

## AI Agent 用户（一等公民）

> 这是本产品区别于一切传统 PM 工具的根本。Agent 不是"功能"，是**用户**——它读、它写、它维护。系统的数据格式、接口、契约都必须为"agent 能直接消费/产出"而设计。

### A1 · 上游写入 Agent

| 维度 | 内容 |
|---|---|
| **角色** | 企微/飞书机器人、各类采集 agent。 |
| **要做的事(JTBD)** | "把人在 channel 里说的需求/反馈，结构化地、带上下文地写进需求池。" |
| **服务它的要求** | 系统要有**机器可写的入口 + 受控的结构**，让 agent 写入即合规、即可被聚合。 |
| **替谁分忧** | 替 P2 的"不会主动维护"兜底——把人力采集变成 agent 采集。 |

### A2 · 下游消费 Agent

| 维度 | 内容 |
|---|---|
| **角色** | 做 planning、产品设计、implementation、测试的 agent。 |
| **要做的事(JTBD)** | "直接读取机器可读的 feature 定义/需求/依赖，产出 planning 草案、设计、代码、测试，再写回。" |
| **服务它的要求** | 产物必须**机器可读（Markdown+YAML/结构化）**、有明确契约和依赖边，agent 才能可靠消费。 |
| **替谁提速** | 替 P1/P3 把"链路 AI 化"，打破人力瓶颈（对治 R5）。 |

### A3 · 维护治理 Agent

| 维度 | 内容 |
|---|---|
| **角色** | 持续跑在后台的看护 agent。 |
| **要做的事(JTBD)** | "持续给需求池保鲜（去重、降级过期需求、补全细节）、做漂移检测、影响分析、提议合并。" |
| **服务它的要求** | 系统要把"治理动作"暴露成 agent 可执行的操作，并通过提议/审批门禁落地。 |
| **替谁兜底** | 替 R3（熵增）和 R4（漂移）做系统性、持续性的看护——这正是人做不到的部分。 |

## 跨画像的统一诉求（最深层）

把五类人 + 三类 agent 的诉求向下钻到底，会汇聚到**同一个核心诉求**：

> **"让产品的真相只有一处、永远可信、人和 AI 都能读写——这样我才能把复杂度交给系统、把重复交给 agent，自己只做判断。"**

这条统一诉求，正是 [产品需求.md](产品需求.md) 全部能力需求的源头，也是 [产品痛点.md](产品痛点.md) 全部根因的镜像。

| 谁 | 向下钻到底的那句话 |
|---|---|
| P1 掌舵者 | 我要随时可信地知道全局状态，并能放心授权。 |
| P2 供给者 | 我说过的，别丢。 |
| P3 执行者 | 给我当前、准确、无歧义的定义。 |
| P4 交付者 | 让定制和手册不再失控漂移。 |
| A1/A2/A3 | 给我机器可读、可写、有契约的底座，我来接管重复劳动。 |

---

## 设计取舍提示（给后续需求）

1. **P1 是核心，P2 是"绝不能指望其自律"的约束。** → 采集必须自动化（A1 承接），不能寄望 P2 填表。
2. **Agent 与人共享同一份数据，而非两套。** → 一份 Markdown+YAML，人读正文、机读头，禁止"给人一套、给机器一套"。
3. **服务 agent ≠ 全自动。** 口述明确"人和 AI 协作并存"。后果性改动仍需人审（呼应仓库的 PR/risk_tier 门禁）。
