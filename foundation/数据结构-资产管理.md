# 数据结构 · 资产管理

> **目的：** 从《仿真平台资源汇总》Excel（16 个工作表）抽象出**到底管了哪些资产、需要哪些数据结构来描述它们**，作为新平台的数据模型设计输入。
>
> **一句话结论：** 16 张表不是 16 种孤立结构，而是**一个"资产基类 + 多态子类"的继承体系**：所有资产共享一套核心字段（身份/分类/适用性/版本/Tag/引擎引用），各类型再叠加自己的特有字段。其中**有 2 张表根本不是资产**（评价指标、标准分类），应单列实体。

---

## 3. 资产基类（Asset Base）—— 所有资产共享的核心字段

剖析 6 张主资产表后，提炼出稳定复现的公共字段骨架：

| 字段组 | 字段 | 类型 | 枚举/约束 | 说明 |
|---|---|---|---|---|
| **身份 Identity** | name_cn | string | 必填 | 资源-中文 |
| | name_en | string | 命名规范见 note | 资源-英文（基线要求 `下划线+序号`，如 `Man_Squat_01`） |
| | asset_id | id | 唯一 | asset_id / MapId / Pak_ID（各表叫法不一） |
| **分类 Category** | category | enum(受控) | 各子类自己的细分树 | 细分-中英文⭐ |
| | ue_category | enum | 一级/二级 | UE类别（引擎侧归类） |
| **媒体 Media** | thumbnail / topview | image | 未提取 | 缩略图/俯视图（源为嵌入图） |
| **适用性 Applicability** | by_edition | map<edition,bool> | 见 §4 | 公有云/社区版/协作版/企业版(单机/云端)/教育版(教学/科研) |
| | by_project | map<project,bool> | 见 §4 | 项目-毫末/清华/均胜/零束/国创/地平线… |
| | baseline | bool | **现以行底色表达，应显式化** | 是否纳入基线 |
| **版本 Version** | impl_version | semver | 2.0.0 起 | 实现版本 |
| | update_version | semver | | 更新版本 |
| | project_version | string | | 项目版本⭐ |
| **标签 Tag** | scene_tags | [string] | | 场景Tag⭐ |
| | asset_tags | [string] | | 资产Tag⭐ |
| **引擎引用 EngineRef** | ue_name | string | | UE_名称 |
| | ue_path | string | | UE路径 |
| | level 所属关卡 | string | | 所属关卡 |
| | collision_box | bool | □/■ | 制作碰撞包围盒 |
| **几何 Geometry** | size_cm | string | `L*W*H` | 尺寸（cm） |
| **审计/备注** | note | text | | 备注（常含曾用名、命名规则、删改记录） |

> 这套骨架 = 资产的"presence/version/tag/实现引用"四件套，**与功能侧的适用性/版本模型同构**——说明功能与资产应共用同一套"适用性 + 版本 + Tag"底层原语。

## 4. 适用性维度：两条正交的轴（含关键数据质量问题）

剖析发现资产的适用性由**两条独立的轴**表达，且都存在不一致：

### 轴 A：按商业版/形态（edition）
取值在各表间**严重不一致**——这是必须收敛的受控词表问题：

| 表 | 出现的 edition 列 |
|---|---|
| 主车 | 公有云/社区版/企业版-单机/企业版-云端/教育版(含教学)/企业版⭐科研版/同济 |
| 动态/静态 | 公有云/社区版/协作版/企业版(云端/单机)/教育版(含教学) |
| 传感器 | 公有云/社区版/协作版/企业版-单机/企业版-云端 |
| 动力学 | + 协作版-客户端/协作版-云端/教学版(参考协作版)/航四 |
| 环境/判定/特殊区域/交通流 | 仅 企业版⭐科研版 / 教学版（粒度粗很多） |

> **设计要求 D1（与功能侧同款）：** edition 适用性必须来自统一 `dimensions.yaml` 的 variant_cells，禁止每张表各立一套。当前至少 4 种粒度并存。

### 轴 B：按项目（project）
以一组**临时加的项目列**表达：`项目-毫末`/`项目-清华模拟器`/`项目-均胜`/`项目-零束`/`项目-国创大赛`/`人工智能大赛`/`地平线二期`/`劢擎港机`/`航四`/`同济`…

> **设计要求 D2：** 项目适用性不应是"每来一个项目就加一列"，而应是 `Asset × Project` 的**关系边**（与功能侧 ProjectCustomization 复用同一个 Project 实体）。否则列会无限膨胀（痛点 R2/R3 的微观重演）。

### 轴 C（隐藏）：基线归属用"行底色"表达
源表注释明确"整行高亮=是否纳入基线"（绿=基线/蓝=未纳入）。

> **设计要求 D3：** 基线归属必须是**显式字段** `baseline: bool`，不能靠颜色——颜色不可机读、不可追溯（痛点 R5）。

## 5. 各子类的特有字段（多态扩展）

基类之外，每个资产子类叠加自己的专有字段：

### Map 地图（34 列，最重）
`OD(xodr文件名)` · `MapId` · `程序关卡/美术场景` · `涉及交通灯` · `夜景灯光关卡 NightLevel` · `是否适配雨雪` · `感知地面标线` · `虚实线` · `红绿灯` · `语义` · `碰撞` · `Nanite资源情况` · `旋转180` · `logo露出` · `是否公开` · `Pak_ID`
> 注：地图的 A 列是 **xodr 文件名（身份）**，细分在 D 列——与其它表"A=细分"不一致（数据质量问题 D4）。

### Vehicle 主车（28 列）
`俯视图` · `主车后轴中心到box几何中心` · `车灯与大灯` · `制作碰撞包围盒` · `UE类别(一/二级)`
> 子类型由细分区分：主车模型（带几何/UE）vs 主车预设（算法配置，如"自动驾驶-决策规划，用于 Apollo 测试"）。

### DynamicElement 动态元素（40 列，字段最多）
`交通流类别(标准规程/SOTIF 不进交通流)` · `颜色标签` · `类型(car/…)` · `做旧` · `曾用名(中/英)` · **合成数据组**：`Syth Tag`/`Syth Ready`/`2D标注`/`3D标注`
> 合成数据(Synthetic Data)是动态/静态元素特有的一组标注就绪度字段，指向 Dataverse 侧用途。

### StaticElement 静态元素（34 列）
`资源包🛒` · `做旧` · 合成数据组（`Syth Tag`/`Syth Ready`）· `类型`

### Sensor 传感器（33 列，最"硬件"）
**型号信息组**：`型号-软件配置项` · `模组/相机整机型号` · `图像传感器芯片` · `镜头型号` · `水平视场角HFOV` · `分辨率` · `Source` · `适用场景`
> 传感器更像"硬件型号台账"，特有字段几乎不与其它资产重叠——子类差异最大。

### Preset 预设族
- Environment：天气预设（晴天-早晨/中午…）
- Dynamics：第三方动力学（CarMaker/CarSim，含"3.8版本移除"等生命周期）
- Judgement：原子判定（停车/启停）
- 其余为轻量分类+适用性。

## 6. 非资产实体（必须单列）

### EvaluationMetric 评价指标（64 条）
| 字段 | 源列 | 说明 |
|---|---|---|
| no | 编号 | |
| dimension | 评价维度 | 安全(39)/舒适性(9)/违规(6)/稳定性(4)/准确性(3)/能耗(2)/效率(1) |
| metric | 评价指标 | 如"变道最大横向加速度" |
| definition | 指标特征定义 | 计算口径 |
> 这是**测试/评测域**的概念，不是仿真资产。应独立建模，供"测试管理"功能消费。

### StandardTaxonomy 标准分类（17 条）
`设施大类` · `设施` · `定义` · `别名` · `基本形式` · `来源(NAS路径)`
> 行业设施标准参考（道路作业安全设施…），是**外部受控词表来源**，用于规范静态元素的分类，不是资产实例本身。

## 7. 抽象出的目标数据结构（汇总）

```yaml
Asset:                          # 资产基类（多态）
  asset_id: string              # 唯一
  type: enum                    # Map|Vehicle|DynamicElement|StaticElement|Sensor|RoadsideEquipment|Preset
  name_cn: string
  name_en: string               # 命名规范受控
  category: enum                # 子类各自的细分树
  media: {thumbnail, topview}   # 图，单独存储
  applicability:
    by_edition: {variant_cell: bool}     # ← 统一 dimensions.yaml（D1）
    baseline: bool                        # ← 显式字段，非颜色（D3）
  projects: [Asset×Project 边]            # ← 关系边，非动态列（D2）
  version: {impl, update, project}
  tags: {scene: [], asset: []}
  engine_ref: {ue_name, ue_path, level, collision_box}
  geometry: {size_cm}
  note: text
  # —— 子类特有字段（多态）——
  # Map:    {od_file, map_id, traffic_light, night_level, rain_snow, semantic, nanite, pak_id, ...}
  # Sensor: {model, chip, lens, hfov, resolution, source, scene, ...}
  # Dynamic/Static: {synthetic: {syth_tag, syth_ready, label_2d, label_3d}, aging, color, ...}
  # Vehicle: {rear_axle_to_box, headlights, ue_category, ...}

EvaluationMetric:               # 评价指标（非资产）
  no, dimension(enum), metric, definition

StandardTaxonomy:               # 标准分类（受控词表来源）
  facility_class, facility, definition, alias, basic_form, source

Project:                        # 与功能侧共用同一实体
  id: 毫末|清华|均胜|零束|地平线|…
```

## 8. 与功能侧 & 仓库模型的关系

| 概念 | 资产侧 | 功能侧 | 仓库现状 |
|---|---|---|---|
| 适用性维度 | by_edition (variant_cell) | presence (variant_cell) | **同源** → dimensions.yaml |
| 项目归属 | Asset×Project 边 | ProjectCustomization | Project 实体待建 |
| 版本 | impl/update/project_version | revision | 同构 |
| 基线归属 | baseline(行色→字段) | scope=核心 | 需显式化 |
| Tag | scene/asset tags | capabilities | 可并入 taxonomy 标签体系 |

**核心判断：功能与资产共享同一套底层原语**（受控变体词表 + 适用性 + 版本 + 项目边 + Tag）。新平台应把这套原语下沉为**通用资产/对象模型**，功能、资产、未来的测试用例都是它的实例化——这正好兑现口述里"系统资产管理也要纳入统一管理""跨产品通用"的诉求。

## 9. 数据质量问题清单（作为设计约束）

| # | 问题 | 设计要求 |
|---|---|---|
| D1 | edition 词表 4+ 套粒度并存 | 统一受控 variant_cells |
| D2 | 项目用"动态加列"表达 | 改为 Asset×Project 关系边 |
| D3 | 基线归属藏在行底色里 | 显式 `baseline` 字段 |
| D4 | 地图 A 列语义与他表不一致（文件名 vs 细分） | 统一身份/分类列定义 |
| D5 | 中英文命名混用、序号格式不一 | 命名规范进受控校验 |
| D6 | asset_id/MapId/Pak_ID 多套标识 | 统一主键策略 |
| D7 | 含"（删）""3.8移除"等生命周期混在数据里 | 显式 lifecycle 状态字段 |
