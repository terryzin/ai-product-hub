# 产品痛点 · 深度根因分析

> **方法论：** 不停留在"工具散、维护累、矩阵复杂"这些**表象**上。表象是症状，真正要根治的是底下的**结构性根因**。本文把口述里十几条具体抱怨，收敛成 **5 个根因**，并揭示它们如何**互相放大**成一个恶性循环——以及为什么这个循环在 AI 时代会致命。

---

## 1. 痛点全景：从现象到根因

口述里出现的所有具体痛点，先平铺，再各自归因（根因编号见 §2）：

| # | 表层现象（用户原话语义） | 直接影响 | 根因 |
|---|---|---|---|
| 1 | 当前状态"说不太清楚"，到底实现到哪、是什么状态讲不清 | 决策没有事实依据 | R1 |
| 2 | 工具散落 Pingcode / 腾讯文档 / Git Markdown，"整个链条都是断开的" | 信息发散、对不上 | R1 |
| 3 | 需求来源极多（售前/销售/产品/客户反馈/前瞻/项目/老板），收不拢 | 需求在源头就漏 | R3 |
| 4 | 需求"一阵一阵"、有时效性，过了这阵优先级陡降 | 需求池腐化、决策失据 | R3 |
| 5 | 需求池靠腾讯文档人工维护、"凭大家主动性"，聊过的丢细节 | 采集即失真 | R3 |
| 6 | 多维矩阵爆炸：业务域 × 标准/项目 × 形态 × OS × 垂直版本 | 无人能掌握全局 | R2 |
| 7 | 基线 ↔ 项目：哪些合回、哪些不合、哪些合入，"讲不清楚" | 双向 merge 地狱 | R2 |
| 8 | feature 颗粒度可大可小，写文档时"把其他 feature 也覆盖掉了" | 边界混乱、重复 | R2 |
| 9 | 某项目增强了功能，别处对该 feature 的引用/设计文档无法同步更新 | 文档漂移、失真 | R4 |
| 10 | 用户手册跟不上功能：基线一套手册，项目改了内容却同步不了 | 交付质量塌方 | R4 |
| 11 | 发布阶段：基线发了什么、项目发了什么、闭环不清 | 不可追溯 | R1 |
| 12 | 一切产物"面向人"（给人看、给测试看），AI 时代节奏跟不上 | 范式落后 | R5 |
| 13 | 内置资产（地图/动静态/系统预设）多类型、多版本，无处管理 | 管理对象有缺口 | R1\* |

\* 资产问题本质也是"没有纳入统一事实源"，归并入 R1。

## 2. 五大根因（深度展开）

> 关键判断：上面 13 条现象**不是 13 个问题**，而是 **5 个根因**反复投影出的影子。逐个根治根因，现象会成片消失。

### R1 · 没有单一事实源（SSOT 缺失）——"状态不可知"

**本质：** 产品的真相被切成碎片散在多个工具里，没有任何一处是"权威的当前状态"。于是**没有人、也没有 AI 能回答"现在到底是什么状态"**。口述里最扎心的一句——"说不太清楚当前到底是一个什么样的状态"——就是这条根因的直接供词。

**为什么致命：** 一切下游决策（做什么、合不合、发什么、手册怎么写）都建立在"知道现状"之上。现状不可知，则整条链上的每个决策都是在猜。现象 1/2/11/13 全部由此而来。

**旧范式里的错误假设：** "把东西写在某个文档里 = 管起来了。" 真相是：写在 N 个互不连通的地方 = 没管起来，反而制造了"看起来有、其实对不上"的假象。

### R2 · 用复制与 fork 表达差异（缺参数化模型）——"复杂度爆炸"

**本质：** 产品的复杂度本身是客观的（多业务域 × 标准/项目 × 形态 × OS × 垂直版本），但**旧范式用"复制一份再改"来表达每一种差异**。复制是线性投入、却带来组合级的维护负担。基线和项目之间用类 git-branch 的方式分叉，于是"哪些合回/哪些不合/哪些合入"变成一道永远算不清的题。

**为什么致命：** 复杂度不会消失，但表达方式决定它是"可管理"还是"地狱"。复制式表达让差异**无法机检、无法聚合、无法一眼看全局**；feature 颗粒度也因此失控——因为没有"定义一次"的锚点，写新文档时自然会覆盖旧的。现象 6/7/8 同源。

**旧范式里的错误假设：** "不同的版本/项目就该有不同的文档。" 真相是：差异应当是**同一个定义上的增量（参数/override/profile）**，而不是另起一份。

> 📌 本仓库已对此有雏形：features 的 `presence`（适用性）+ `override`/`parameter`（差异）+ `product-lines/*.yaml`（profile 分支）+ 不可变 `revision`（血缘）。R2 是仓库现有设计最直接命中的根因。

### R3 · 维护依赖人的自律，而非系统保证——"熵增与失真"

**本质：** 需求的采集与保鲜，今天靠"人记得去录、记得更新、记得维护优先级"。口述原话："凭大家主动性"、"聊的东西没及时进，或即便进了也丢了很多细节"。**任何依赖人持续自律的流程，都会随时间腐化**——这是熵增定律，不是态度问题。

**为什么致命：** 需求是持续涌入、且有时效性的流。靠人维护的池子，必然出现"采集即失真 + 越积越乱 + 优先级过期"三重腐化，最终池子里的数据不再可信，决策也就失去依据。现象 3/4/5 同源。

**旧范式里的错误假设：** "建个表 + 大家自觉填 = 需求池。" 真相是：**采集必须零摩擦/自动化，治理必须是系统的持续动作**，否则池子注定变垃圾场。

### R4 · 一致性靠人维护，而非"构造即一致"——"血缘断裂与文档漂移"

**本质：** feature、它的设计文档、对它的引用、对应的用户手册，今天是**四份各自独立的东西**，靠人记得"改了 A 要去改 B/C/D"。一旦某处（尤其项目分支）改动，其余三份立刻漂移。口述里"怎么保持 up to date"被反复追问，正是这条根因。

**为什么致命：** 文档一旦不可信，就退化为"没有"——甚至比没有更糟，因为它会误导人和 AI。用户手册跟不上功能（现象 10）是这条根因最贵的后果：直接砸在客户交付上。现象 9/10 同源。

**旧范式里的错误假设：** "改了记得同步就行。" 真相是：一致性必须是**系统的构造性属性**——手册/引用应当从 feature 的单一定义派生或链接血缘，而不是平行维护。

### R5 · 范式面向人，而非面向 AI——"在 AI 时代结构性落后"

**本质：** 过去所有产品产物（文档、需求、设计）都是**写给人看的**——给开发看、给测试看。但工作节奏已进入 AI 时代，瓶颈从"人的产出"变成"能否让 agent 源源不断地接手"。口述里点破："可能未来我的文档就不是写给人看了，完全写给机器看其实就可以了。"

**为什么致命：** 这不是"加个 AI 功能"的增量问题，而是**底座范式问题**。如果产物不是机器可读、不能被 agent 直接读写消费，那么需求采集、planning、产品设计、实现、测试就无法接入 agent 流水线，迭代速度被人力死死锁住。现象 12 是表象，真正的代价是"整条链路无法 AI 化提速"。

**旧范式里的错误假设：** "AI 是用来辅助人的工具。" 真相是：**AI agent 是一等公民（既是上游写入者、也是下游消费者、还是维护者）**，系统必须原生为"人 + agent 双层用户"设计。

---

## 3. 根因之间不是并列，而是一个恶性循环

五大根因互相喂养，形成自我强化的下滑螺旋：

```
R1 状态不可知
   ↓ 因为看不清全局……
R2 只能靠复制/分叉硬扛复杂度
   ↓ 复制越多，需要维护的点越多……
R3 维护超出人的自律极限 → 熵增
   ↓ 维护跟不上……
R4 文档/手册/引用集体漂移 → 真相进一步失真
   ↓ 真相更不可知，回到……
R1（更严重）
   ↓ 整个循环“面向人”手工运转，无法交给 agent 提速……
R5 在 AI 时代被指数级放大
```

**核心结论：** 单点修补（比如"再买个工具""再建个表"）只会被这个循环吞掉。必须从 **R1（建立 SSOT）** 切入打断循环，再用 **R2（参数化建模）+ R4（构造性一致）** 让复杂度可控、一致性自动，用 **R3（零摩擦采集 + 持续治理）** 止住熵增，最后用 **R5（AI-native 底座）** 把整条链路交给 agent 提速。

---

## 4. 如果不解决：在 AI 时代代价被放大

- **人会成为唯一瓶颈：** 当别人用 agent 把需求→实现的链路自动化，仍"面向人手工维护"的团队，迭代速度差距会被指数拉开。
- **复杂度持续复利：** 业务域还在扩张（自驾 → 无人船/机/矿/码头/具身），矩阵只会更大；不换表达范式，维护负担是复利增长。
- **交付质量塌方：** 手册/文档漂移直接命中客户交付，且项目交付后客户在自己分支迭代、不回升，漂移永久化。
- **知识无法沉淀为资产：** 聊过即失真，团队的产品知识无法累积成可被 AI 复用的底座，每次都从零开始。
