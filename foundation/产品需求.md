# 产品需求 · 深度推导

> **方法论：** 不抄口述里的功能名词（"需求池""Dashboard""资产管理"）。那些是用户**自己设想的解法**。本文做两件事：①把每个解法**还原回它要满足的深层诉求**；②从诉求出发**重新推导**出能力级需求，确保我们解决的是真问题，而不是照着别人画的 UI 施工。
>
> 每条需求都回链到 [产品痛点.md](产品痛点.md) 的根因(R\*) 和 [用户画像.md](用户画像.md) 的画像(P\*/A\*)。

---

## 0. 一条产品原则（凌驾于所有需求之上）

口述里所有诉求向下钻，汇聚成一句**北极星**：

> **让产品的真相只有一处、永远可信、人与 AI 都能读写。**

由此推出三条不可违背的设计原则，后面每条需求都必须服从：

| 原则 | 含义 | 反面（禁止） |
|---|---|---|
| **单源** | 每个事实只定义一次，差异用增量表达 | 复制一份再改 / fork 分支 |
| **构造性一致** | 一致性由系统结构保证，而非人去同步 | "改了记得同步" |
| **人机同源** | 人和 agent 读写同一份数据 | 给人一套、给机器一套 |

## 1. 需求全景图

口述把工作分成"需求 → planning → 开发 → 发布 → 反馈"五段闭环，外加资产、Dashboard、AI-native。我把它们重组为 **7 个能力群**（R1–R7），其中 R7 是横切全局的底座：

```
            ┌──────────────────────── R7 AI-native 底座（横切）────────────────────────┐
            │                                                                          │
  上游      │   R3 需求池          R5 规划           R4 一致性         R6 发布          │   下游
  channel ──┼─▶（采集+保鲜）──▶（抽象+分配）──▶（feature 单源+血缘）──▶（版本+闭环）──┼─▶ 交付
            │        ▲                                  │                              │
            │        └──────────── 反馈闭环 ◀───────────┘                              │
            │                                                                          │
            │            R1 单一事实源（一切的地基） · R2 变体与分支建模 · 资产纳管        │
            └──────────────────────────────────────────────────────────────────────┘
```

## 2. 能力群（深度推导）

### R1 · 单一事实源（SSOT）—— 地基

**用户的解法说法：** "要一个大的数据库，把需求/文档/实现都管起来，集中管理。"

**还原深层诉求：** 用户要的不是"数据库"这个技术物，而是 **"任意时刻能可信地回答：某 feature 现在是什么状态、在哪些产品线、谁依赖它、历史如何"**。核心词是**可信**与**唯一**。

| 能力需求 | 说明 | 服务 |
|---|---|---|
| R1.1 一功能一事实源 | 每个 feature 一处权威定义（机器头 + 人读正文合一） | R1 / P1,P3 |
| R1.2 状态可问答 | 能就"当前状态/实现进度/所在线/依赖"即时给出可信答案 | R1 / P1 |
| R1.3 历史可追溯 | 任何改动留痕，能回看"当初是什么样" | R1,R4 / P1,P3 |
| R1.4 万物归位 | 需求/文档/实现/资产都挂到这棵树上，不再散落 | R1 / P1 |

> **仓库现状：** features/ + YAML 头已是雏形锚点；缺"状态问答"和"实现进度回填"。
> **价值排序：P0。** 这是打断恶性循环的入口，其它能力都站在它之上。

---

### R2 · 变体与分支建模 —— 让复杂度可控

**用户的解法说法：** "管好基线和项目分支，哪些合回、哪些不合……还有单机/云端/协作 × Win/Linux × 标准/教育/科研 × 多车 的矩阵。"

**还原深层诉求：** 用户**不是要一个 merge 工具**——他要的是**让差异不再靠复制表达**，从而"基线↔项目""产品矩阵"不再是算不清的题。深层诉求是：**差异 = 同一定义上的可机检增量**。

| 能力需求 | 说明 | 服务 |
|---|---|---|
| R2.1 适用性建模 | 标"哪些变体单元适用/例外"，而非给每个组合复制一份 | R2 / P1,P4 |
| R2.2 差异即增量 | 行为差异用参数/override 表达，并能继承，不复制 | R2 / P4 |
| R2.3 产品线=profile | 客户线/版本=勾选+override+排除的 profile，**禁止 git fork** | R2 / P1,P4 |
| R2.4 归属决策 | 一个需求是"核心/可晋升/客户独占"，把双向 merge 塌缩成一次性归属判断 | R2 / P1,P4 |
| R2.5 颗粒度锚定 | 有"定义一次"的锚，feature 边界清晰、不互相覆盖 | R2 / P1,P3 |

> **仓库现状：这是仓库现有设计命中最准的一群**——presence / override / parameter / product-lines profile / scope 已全部就位。R2 主要是"用起来、补全 by_cell"。
> **价值排序：P0（建模），但仓库已有雏形，重点在落地。**

---

### R3 · 需求池全生命周期 —— 采集零摩擦 + 系统化保鲜

**用户的解法说法：** "要个需求池数据库，接企微/飞书机器人自动进，还要持续管理、避免熵增。"

**还原深层诉求：** 关键洞察来自痛点 R3——**人不会主动维护**。所以这条需求的真正命题不是"建池子"，而是 **"让采集不依赖人的自律、让保鲜成为系统/agent 的持续动作"**。

| 能力需求 | 说明 | 服务 |
|---|---|---|
| R3.1 零摩擦采集 | 经 channel（企微/飞书 + agent）自动入池，不要求 P2 学工具/填表 | R3 / P2,A1 |
| R3.2 带上下文不失真 | 入池保留原始上下文与细节，"聊过的别丢" | R3 / P2,A1 |
| R3.3 按 feature 组织 | 需求挂到 feature/能力上，而非一张平表 | R3,R1 / P1 |
| R3.4 抗熵保鲜 | 去重、补全、给过期/低时效需求**自动降级**，池子长期可信 | R3 / P1,A3 |
| R3.5 决策数据支撑 | 每条需求带足够元数据（来源/域/时效/优先级）支撑 planning | R3,R5 / P1 |

> **仓库现状：空白。** 这是当前仓库最缺、却最贴近痛点根因的一群。
> **价值排序：P1（紧接 SSOT 之后，且是 agent 价值最早兑现处）。**

---

### R4 · 一致性与血缘 —— 文档/手册不再漂移

**用户的解法说法：** "怎么保持 feature/文档/用户手册 up to date？项目改了，别处引用和手册同步不了。"

**还原深层诉求：** 用户要的不是"提醒大家同步"，而是 **"让一致性成为系统的构造性属性"**——手册、引用从 feature 单源**派生或链接血缘**，改一处，相关处要么自动随动、要么被精确标记为待更新。

| 能力需求 | 说明 | 服务 |
|---|---|---|
| R4.1 不可变版本 | 改动=新版本，旧版本冻结可追溯，不原地改 | R4 / P1,P3 |
| R4.2 依赖血缘 | feature 间依赖成 DAG，改 B 能反查谁受影响（影响分析） | R4,R2 / P1,P3 |
| R4.3 漂移检测 | 自动发现"定义改了但引用/手册没跟上"，标记待更新 | R4 / P1,A3 |
| R4.4 手册随功能走 | 用户手册与 feature 血缘绑定，功能变→手册被标记/派生更新 | R4 / P4 |
| R4.5 项目交付后不失联 | 客户分支持续迭代的场景下，仍能追踪其与基线的血缘差 | R4,R2 / P4 |

> **仓库现状：** 不可变 revision + dependencies + pin 已是雏形（R4.1/R4.2）；漂移检测、手册随动是空白。
> **价值排序：P1。手册随动（R4.4）是"交付质量"层面的高价值点。**

---

### R5 · 规划（planning）—— 从需求到可落地功能点

**用户的解法说法：** "对汇总的需求做全局分析抽象，变成可落地功能点，并判断进项目还是进产品迭代。"

**还原深层诉求：** 这是**人 + agent 协作的判断环节**。诉求是 **"在可信的需求池之上，做抽象与分配决策，且 agent 能起草、人来定夺"**。

| 能力需求 | 说明 | 服务 |
|---|---|---|
| R5.1 需求→功能点抽象 | 把零散需求归并抽象为 feature/能力级条目 | R5 / P1,A2 |
| R5.2 分支适配决策 | 判断进基线还是进项目、是否可晋升（接 R2.4 归属） | R5,R2 / P1 |
| R5.3 优先级与时效 | 结合产品/项目/技术迭代状态排优先级 | R5,R3 / P1 |
| R5.4 agent 起草+人审 | agent 出 planning 草案，人做后果性决策 | R5,R7 / A2,P1 |

> **仓库现状：空白（治理门禁 proposals/ 是雏形）。**
> **价值排序：P2（依赖 R3 先有可信池子）。**

---

### R6 · 发布与闭环 —— 可追溯 + 反馈回流

**用户的解法说法：** "管理发布版本，知道基线/项目各发了什么，做闭环；发布后反馈回到需求池。底层接 Pingcode 双向同步。"

**还原深层诉求：** 诉求是 **"让'发了什么'可追溯、让反馈自动回流形成闭环，并与执行系统(Pingcode)对接而非取代它"**。

| 能力需求 | 说明 | 服务 |
|---|---|---|
| R6.1 发布版本管理 | 清晰记录基线/各项目发布了哪些 feature/版本 | R6,R1 / P1,P4 |
| R6.2 任务闭环 | feature 开发→发布闭环可视、可追溯 | R6 / P1 |
| R6.3 反馈回流 | 发布后用户反馈自动回需求池，闭合链条 | R6,R3 / P1,A1 |
| R6.4 与 Pingcode 双向同步 | 作为上游汇总，向 Pingcode 下发、并读回实现状态 | R6,R1 / P1,A2 |

> **关键边界：** 用户明确"底层项目/任务管理接 Pingcode"。**本系统不取代 Pingcode**，而是做其上游的"产品事实源 + 汇总层"。这是重要的非目标，见 §4。
> **价值排序：P2。**

---

### R7 · AI-native 底座（横切，凌驾全局）

**用户的解法说法：** "整个系统要 AI native，能接各种 channel，agent 能读能写能维护。"

**还原深层诉求：** 这不是一个"功能群"，而是 **R1–R6 每一条都必须满足的底层契约**：数据机器可读、接口 agent 可调、操作可被 agent 执行且受治理门禁约束。它对治痛点 R5（范式落后）。

| 能力需求 | 说明 | 服务 |
|---|---|---|
| R7.1 机器可读产物 | 所有事实以结构化文本（Markdown+YAML/受控词表）承载 | R5(痛点) / A1,A2,A3 |
| R7.2 上游可写接口 | channel/agent 能合规写入（采集、提议） | / A1 |
| R7.3 下游可消费接口 | agent 能可靠读取定义/依赖去做 planning/实现/测试 | / A2 |
| R7.4 治理可执行 | 保鲜/漂移/影响分析等治理动作 agent 可执行 | / A3 |
| R7.5 人机协作门禁 | 后果性改动经提议→人审落地（risk_tier） | / P1,A3 |
| R7.6 跨产品通用 | 不耦合单一产品，SimOne/Dataverse/未来产品通用 | / P1 |

> **仓库现状：** Markdown+YAML 底座 + PR/risk_tier 门禁是雏形（R7.1/R7.5）；channel 接入、agent 接口是空白。
> **价值排序：P0 作为原则贯穿；具体接口实现 P1–P2 渐进。**

## 3. 优先级总览（建议路线）

> 排序逻辑：先打断恶性循环的**地基**（R1/R2），再兑现**最贴痛点根因、agent 价值最早显现**的部分（R3/R4），最后补**判断与闭环**（R5/R6）。R7 全程作为底层契约。

| 阶段 | 能力群 | 为什么是现在 |
|---|---|---|
| **P0 地基** | R1 单源 · R2 变体建模 · R7 底座原则 | 不先有可信单源 + 参数化模型，后面全是沙上建塔。**仓库已对 R1/R2 有雏形，先夯实。** |
| **P1 止血** | R3 需求池保鲜 · R4 一致性/血缘 | 直击熵增(R3)与漂移(R4)两大根因；也是 agent（A1/A3）价值最早兑现处。 |
| **P2 闭环** | R5 规划 · R6 发布闭环 | 依赖前两阶段的可信数据；把链路接成闭环并对接 Pingcode。 |
| **横切** | 资产纳管 | 作为 R1 的一类对象纳入（地图/动静态/系统预设，分版本），不单列阶段。 |

## 4. 边界与非目标（同样重要）

明确**不做什么**，避免范围蔓延：

| 非目标 | 原因（口述依据） |
|---|---|
| **不取代 Pingcode** | 用户明确项目/任务/发布底层接 Pingcode；本系统是其**上游产品事实源 + 汇总层**，双向同步而非替代。 |
| **不取代沟通工具** | 采集走企微/飞书既有 channel，不把 P2 请进新工具。 |
| **不追求全自动** | 口述强调"人和 AI 协作并存"；后果性改动必须人审。 |
| **不与单一产品耦合** | 必须跨产品（SimOne/Dataverse/未来）通用，不写死 SimOne 概念。 |
| **不用 git branch 表达客户线** | 用 profile 表达，避免双向 merge 地狱（呼应 R2.3）。 |
